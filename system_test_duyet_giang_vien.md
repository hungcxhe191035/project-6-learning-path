# System Test Cases - Lecturer Approval Workflow

This document specifies the System Test Cases for the **Instructor Application & Approval Workflow** in the system. The test suite is designed in accordance with the 15-column table format shown in the reference sheet. It replaces specific URL paths (e.g., `/instructor/apply`) with descriptive screen and menu names, uses standard sequential IDs (`TC_001` to `TC_018`), and provides suggested screenshot evidence for each test case in the **Note** column.

---

## 1. Workflow Summary

1. **Candidate (Student)**: Accesses the **Instructor Registration Form** to input professional info (Headline, Bio, Motivation, LinkedIn URL, and Teaching Areas/Tags) and uploads a CV file in PDF format (maximum size 10MB). Upon submission, the application is stored with a `Pending` status.
2. **System Rules**: Validates security constraints (user must have the `Student` role, cannot have another `Pending` application, the CV must be a `.pdf` file under 10MB). If a previous application was `Rejected`, the system pre-populates the form with the old text and automatically cleans up the old CV file on the server when a new one is submitted.
3. **Administrator**: Accesses the **Lecturer Applications List** and the **Application Details** screen on the Admin dashboard. The Admin downloads/views the PDF CV and makes a decision to **Approve** or **Reject** the application.
   - If **Approved**: The application status changes to `Approved`, the user's role is automatically updated from `Student` to `Instructor`, and the system triggers a web notification and a congratulatory email.
   - If **Rejected**: The Admin must provide a rejection reason in the review notes. The application status changes to `Rejected`, the user's role remains `Student`, and the system triggers a web notification and email explaining the rejection reason.
